Block-STM contention profiler · Aptos Move

Your contract is serializing. Find out where.

Aptos runs a block in parallel and re-executes only what conflicts. But one counter every transaction touches, one pool every swap mutates, and the whole block collapses into a single chain. Movespan names the resource costing you parallelism — and tells you what fixing it is worth, before you refactor anything.

01

Two lines of Move, an order of magnitude apart

Nothing in the toolchain flags the difference. Both compile, both pass tests, both are correct. One of them throws away the reason you chose Aptos.

serializes the block
// every swap mutates the same Pool
public entry fun swap(user: &signer, amt: u64) {
    let pool = borrow_global_mut<Pool>(@amm);
    pool.reserve_x = pool.reserve_x + amt;
    // ^ 199 read-after-write deps
    //   → 1.00x on 32 threads
}
stays parallel
// shard the hot field per user
public entry fun swap(user: &signer, amt: u64) {
    let s = borrow_global_mut<Shard>(shard_of(user));
    s.reserve_x = s.reserve_x + amt;
    // ^ writers stop colliding
    //   → 9.11x on 32 threads
}

03

Observed, not inferred

Access sets come from the actual VM. Static analysis cannot know which dynamic address a transaction will touch, and a wrong access set produces a confidently wrong report.

01 / EXECUTE

Real transactions

Your package is published into an in-memory chain and the workload runs against it — through the real Aptos VM, not a simulation of one.

02 / CAPTURE

Every read and write

A recording state view wraps the executor and observes each read; writes come from the transaction's write set. One transaction per block, so attribution is exact.

03 / MODEL

The conflict graph

Transaction j depends on i when it reads what i wrote. A list scheduler over Block-STM's workers gives work, span, and the parallelism actually available.

04 / ADVISE

Simulate the fix

Rank locations by the work they block, map each to a concrete Aptos refactor, then re-run the whole model with that location neutralized.

Gas payment, sequence numbers and other framework state are excluded on purpose. They touch 100% of every block, so they out-rank whatever your contract actually does wrong — and no contract author can refactor them.

04

Measured, and checkable by hand

The counter fixture is a correctness oracle rather than a benchmark. 100 transactions against one global counter must produce exactly 99 read-after-write dependencies, a fully serial schedule, and a score of exactly 1/8 on 8 threads. It does. That is why the model is trustworthy and not merely plausible.

05

What these numbers are, precisely

A profiler that overstates its own accuracy is worse than no profiler. So, plainly:

  • ExactThe read and write sets. They come from the VM, not from analysis.
  • ExactThe dependency graph and hotspot ranking, and the direction and relative magnitude of a fix.
  • EstimateAbsolute throughput. The scheduler models Block-STM's behaviour but is not a validator. Read 9.11x as "a large win worth doing", not as a TPS guarantee.
  • Not modeledValidator-level effects — network, consensus, storage I/O, and Block-STM's own re-execution and abort overhead.
  • UntestedMainnet replay is implemented but has not yet been exercised against a live fullnode.
06

Run it

The model is a pure Rust workspace with no Aptos dependency — it compiles and tests in under a second. Only the capture layer needs aptos-core. 

# 1 — prove the model (no Aptos build needed)
cargo test --workspace

# 2 — build the profiler (one-time aptos-core build)
cd app && cargo build --release --locked

# 3 — compile a contract and profile it
(cd fixtures/counter && aptos move compile --named-addresses counter=0xC0FFEE)

app/target/release/movespan analyze \
  --package fixtures/counter \
  --module-address 0xC0FFEE \
  --workload workloads/counter.toml

--min-score 0.4 exits non-zero below a threshold, so a pull request that regresses contention fails CI the way an untested one fails a coverage gate.
